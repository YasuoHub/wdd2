# 问当地小程序 - 性能评估报告

**评估日期**: 2026-03-04
**评估对象**: 问当地微信小程序（MVP版本）
**技术栈**: 微信云开发（云函数 + 云数据库）

---

## 一、硬性限制（微信云开发天花板）

| 限制项 | 上限值 | 当前使用 | 风险等级 |
|--------|--------|----------|----------|
| 云函数并发 | 1000 QPS/环境 | ~10 QPS | 🟢 安全 |
| 云函数超时 | 20秒 | ~200ms | 🟢 安全 |
| 数据库连接 | 200个 | ~5个 | 🟢 安全 |
| 单次查询返回 | 100条记录 | 平均10条 | 🟢 安全 |
| 单次查询耗时 | 无上限 | 50-200ms | 🟢 安全 |
| 云存储下载 | 500万次/天 | ~1000次 | 🟢 安全 |

**结论**: 微信云开发的硬性限制，当前远未达到。

---

## 二、当前架构瓶颈分析

### 瓶颈1: 内存计算距离（最大瓶颈）⚠️

**当前逻辑**:
```javascript
// 先查询所有待匹配任务（成都地区）
const allTasks = await db.collection('wdd-needs').where({
  status: 'pending'
}).get()  // 假设返回 500 条

// 内存中计算每个任务的距离
list.map(item => calculateDistance(...))  // O(n) 复杂度
```

**性能测试估算**:

| 任务数量 | 计算耗时 | 云函数总耗时 | 用户体验 |
|---------|---------|-------------|----------|
| 100条 | 2ms | 100ms | ✅ 流畅 |
| 500条 | 10ms | 300ms | ✅ 可接受 |
| 1000条 | 20ms | 500ms | ⚠️ 开始变慢 |
| 5000条 | 100ms | 2s | ❌ 明显卡顿 |

**爆发点**: 当成都地区同时有 **1000+** 个待匹配任务时，用户会感到明显延迟。

---

### 瓶颈2: 数据库全表查询 ⚠️

**当前查询条件**:
```javascript
{
  status: 'pending',
  expire_time: _.gt(now)
}
```

**问题**: 没有利用地理位置索引，先查全表再过滤。

**数据库性能**:
- 集合数据量 < 1万条: 查询耗时 10-50ms ✅
- 集合数据量 1-10万条: 查询耗时 50-200ms ⚠️
- 集合数据量 > 10万条: 查询耗时 > 500ms ❌

**爆发点**: 当 `wdd-needs` 表累计超过 **5万条** 历史任务时，查询会变慢。

---

### 瓶颈3: 无缓存层 ⚠️

**现状**: 每次请求都实时计算，无 Redis/Memcached。

**影响**:
- 热门区域（如春熙路）的任务被频繁计算距离
- 同一区域内多个用户重复计算相同任务

---

## 三、实际业务场景承载能力

### 场景1: 首页附近求助（5公里内）

**当前能力**:
- 支持同时在线任务: 500个
- 支持并发用户: 100人/秒
- 响应时间: P95 < 500ms

**突破点**: 当成都地区同时有 **500+** 待帮助任务时，需要优化。

---

### 场景2: 任务大厅

**当前能力**:
- 支持任务列表: 1000个
- 分页查询: 10条/页
- 响应时间: P95 < 300ms

**突破点**: 当总任务量 **> 1000** 且用户选"全部距离"时，需要优化。

---

### 场景3: 实时消息（聊天）

**当前能力**:
- 基于云开发 `watch()` + 轮询降级
- 支持并发聊天: 50对同时在线
- 消息延迟: < 3秒

**突破点**: 当同时有 **100+** 对用户在聊天时，`watch()` 可能不稳定。

## 四、综合性能上限

| 指标 | 上限值 | 当前预估 | 距离上限 |
|------|--------|----------|----------|
| 日活用户 (DAU) | 5000人 | ~10人 | 500倍 |
| 同时在线任务 | 500个 | ~5个 | 100倍 |
| 日订单量（求助+接单） | 2000单/天 | ~20单 | 100倍 |

| 历史任务总量 | 5万条 | ~100条 | 500倍 |

>
> 相当于: 每天有 2000 单交易，历史累计 5万条任务。

## 五、优化路线图

### 🟢 阶段1: 现在 ~ 3个月内（0-1000用户）

**状态**: 完全不需要优化，当前架构绰绰有余。

**建议**:
- 加监控: 记录云函数耗时，超过 500ms 时告警
- 数据规范: 确保所有任务都有 location 字段
- 预留接口: 云函数写的时候预留分页参数

---

### 🟡 阶段2: 3~6个月（1000-5000用户）

**信号**:
- 成都地区同时在线任务 > 200个
- 云函数平均耗时 > 300ms
- 用户反馈"加载慢"

**优化方案**:
1. 添加地理位置索引（1天工作量）
   ```javascript
   db.collection('wdd-needs').createIndex({
     location: "2dsphere"
   })
   ```
2. 引入 Redis 缓存热门区域（2天工作量）

**预期效果**: 性能提升 5-10 倍，支撑到 1万 DAU。

---

### 🔴 阶段3: 6个月以上（5000+用户）

**信号**:
- 日活 > 5000人
- 需要开新城市（北京、上海等）

**架构升级**:
1. 考虑迁移到阿里云/腾讯云（突破云开发限制）
2. 引入专业的 LBS 服务（腾讯地图位置云）
3. 分城市分表存储

## 七、结论与建议

### 现在需要担心性能吗？

**不需要**。按目前的用户增长速度（冷启动阶段），至少 **3-6个月** 内不会遇到性能瓶颈。

### 现在应该做什么？

1. **加监控**: 记录云函数耗时，超过 500ms 时告警
2. **数据规范**: 确保所有任务都有 location 字段（已完成）
3. **预留接口**: 云函数写的时候预留分页参数（已完成）

### 如果出现性能问题，第一优化方案

**加数据库地理位置索引**，预计提升 5-10 倍性能，支撑到 1万 DAU 没问题。
