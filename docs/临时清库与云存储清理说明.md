# 临时清库与云存储清理说明

本文档说明临时云函数 `wdd-temp-clear-all` 的用途和使用方式。

先说清楚边界：数据库可以通过这个云函数清空非 `wdd-config` 集合；云存储全桶清空不能只靠 `wx-server-sdk` 云函数完成，因为它不能列举整个云存储桶，只能删除已知 `fileID`。如果上线前要求云存储完全为空，必须在微信云开发控制台云存储页面执行全量删除，或使用具备“列举文件”能力的官方工具/API。

## 用途

用于清空问当地小程序测试环境或临时环境的数据：

- 清空业务集合里的文档。
- 保留集合本身和索引。
- 永远保留 `wdd-config` 集合数据。
- 删除业务集合文档中引用到的 `cloud://` 云存储文件。

重要说明：`wx-server-sdk` 可以删除已知 `fileID`，但不能枚举整个云存储桶。因此这个云函数能自动清理“数据库里还能找到引用的文件”，不能保证删除“数据库里已经没有记录的孤儿文件”。首次上线前如果要求云存储完全清空，需要在微信云开发控制台再做一次云存储全量检查和清空。

特别注意：返回 `message: "数据库引用文件分批删除完成"` 只表示当前这一批“数据库里记录过的云文件”删除请求成功，不表示数据库已经清空，也不表示整个云存储桶已经清空。数据库必须继续执行 `mode: "database"`；云存储全桶必须到控制台最终确认。

## 保护规则

1. 默认是预演模式，不删除任何数据。
2. 正式删除必须传确认词：`CLEAR_WDD_DATA_AND_STORAGE`。
3. `wdd-config` 写死保护，即使误传到 `collections` 也会拒绝执行。
4. 数据库只删除文档，不删除集合，因此索引会保留。
5. 如果云存储删除存在失败项，默认停止清数据库，避免文件没删掉但引用记录已丢失。

## 默认清理集合

默认会清理以下集合：

- `wdd-users`
- `wdd-needs`
- `wdd-need-takers`
- `wdd-messages`
- `wdd-notifications`
- `wdd-payment-orders`
- `wdd-balance-records`
- `wdd-point-records`
- `wdd-sign-in-records`
- `wdd-invite-records`
- `wdd-ratings`
- `wdd-reports`
- `wdd-appeals`
- `wdd-tickets`
- `wdd-withdraw-records`
- `wdd-settlement-records`

## 预演调用参数

```json
{
  "dryRun": true,
  "includeStorage": true,
  "expectedEnvId": "wdd-2grpiy1r6f9f4cf2"
}
```

返回结果会告诉你：

- 将要清理哪些集合。
- 每个集合当前有多少条数据。
- 扫描到多少个云存储文件。
- 不会实际删除任何内容。

## 正式清理调用参数

```json
{
  "dryRun": false,
  "includeStorage": true,
  "expectedEnvId": "wdd-2grpiy1r6f9f4cf2",
  "confirm": "CLEAR_WDD_DATA_AND_STORAGE"
}
```

## 如果出现 3 秒超时

如果云端调用日志出现：

```text
Invoking task timed out after 3 seconds
```

说明云函数没有读取到本目录的 `config.json`，云端仍在使用默认 3 秒超时。请重新上传并部署整个 `cloudfunctions/wdd-temp-clear-all` 目录，确认 `config.json` 一起上传。当前配置已把超时时间设置为 60 秒。

如果 60 秒仍然不够，可以分批执行，例如一次只清几个集合：

```json
{
  "mode": "database",
  "dryRun": false,
  "includeStorage": false,
  "expectedEnvId": "wdd-2grpiy1r6f9f4cf2",
  "confirm": "CLEAR_WDD_DATA_AND_STORAGE",
  "collections": [
    "wdd-messages",
    "wdd-needs"
  ]
}
```

分批执行时仍然不能包含 `wdd-config`。

如果控制台云端调用始终显示 3 秒超时，请按下面的 3 秒安全流程执行。

### 第一步：分批清理数据库引用到的云文件

每次只删除少量文件。根据上一次返回的 `data.nextCall.storageOffset` 继续调用，直到 `nextCall` 为 `null`。

第一次调用：

```json
{
  "mode": "storage",
  "dryRun": false,
  "includeStorage": true,
  "expectedEnvId": "wdd-2grpiy1r6f9f4cf2",
  "confirm": "CLEAR_WDD_DATA_AND_STORAGE",
  "storageOffset": 0,
  "storageLimit": 10
}
```

如果仍然超时，把 `storageLimit` 改成 `5`。

### 第二步：分批清空数据库

先清数据量大的集合，每次一个集合。返回结果里如果对应集合 `hasMore` 为 `true`，继续用同样参数重复调用，直到该集合 `remaining` 为 `0`。

示例：

```json
{
  "mode": "database",
  "dryRun": false,
  "includeStorage": false,
  "expectedEnvId": "wdd-2grpiy1r6f9f4cf2",
  "confirm": "CLEAR_WDD_DATA_AND_STORAGE",
  "docDeleteLimit": 100,
  "collections": [
    "wdd-notifications"
  ]
}
```

小集合可以几个一起传，但如果再次遇到超时，就改成一次一个集合。

建议清理顺序：

1. `wdd-notifications`
2. `wdd-point-records`
3. `wdd-payment-orders`
4. `wdd-needs`
5. `wdd-need-takers`
6. `wdd-sign-in-records`
7. `wdd-reports`
8. `wdd-tickets`
9. `wdd-ratings`
10. `wdd-users`
11. `wdd-messages`
12. `wdd-balance-records`
13. `wdd-invite-records`
14. `wdd-appeals`
15. `wdd-withdraw-records`
16. `wdd-settlement-records`

第三步再去云开发控制台的云存储里检查是否还有孤儿文件；如果有，并确认都是测试资源，在控制台清空。

### 查询当前状态

重新部署最新版云函数后，可以用下面参数查看数据库还剩多少条，以及数据库里还能扫描到多少个云文件引用：

```json
{
  "mode": "status",
  "dryRun": true,
  "includeStorage": true,
  "expectedEnvId": "wdd-2grpiy1r6f9f4cf2"
}
```

注意：`referencedFileCount` 是数据库里的 `cloud://` 引用数量。只要数据库还没清，它通常还是原来的数量，不代表云存储文件真的还存在。

### 验证数据库引用的云文件是否还存在

如果想确认前面分批删除的云文件是否真的删掉，可以分批验证：

```json
{
  "mode": "verify-storage",
  "dryRun": true,
  "includeStorage": true,
  "expectedEnvId": "wdd-2grpiy1r6f9f4cf2",
  "storageOffset": 0,
  "storageLimit": 15
}
```

看返回里的 `data.storageVerify.existing`：

- `existing` 为 `0`：这一批数据库引用的云文件已经不存在。
- `existing` 大于 `0`：这一批里还有文件存在，需要重新执行 `mode: "storage"` 删除对应批次，或检查返回里的文件状态。

根据 `data.nextCall.storageOffset` 继续验证下一批，直到 `nextCall` 为 `null`。

## 补充删除孤儿文件

脚本会自动删除数据库文档中能扫描到的 `cloud://` 文件，例如任务图片、聊天图片、头像、举报/申诉图片等。

如果某些云存储文件已经没有数据库记录引用，脚本无法自动发现。可以通过 `extraFileIDs` 手动补充：

```json
{
  "dryRun": false,
  "includeStorage": true,
  "expectedEnvId": "wdd-2grpiy1r6f9f4cf2",
  "confirm": "CLEAR_WDD_DATA_AND_STORAGE",
  "extraFileIDs": [
    "cloud://xxx/need-images/example.jpg"
  ]
}
```

## 首次上线前推荐执行顺序

1. 在微信云开发控制台或其他可信渠道备份当前数据库和云存储。
2. 部署临时云函数 `wdd-temp-clear-all`。
3. 先用预演参数调用，确认目标集合不包含 `wdd-config`，并确认扫描到的文件数量合理。
4. 用正式参数调用，清空业务数据库和数据库引用到的云存储文件。
5. 打开微信云开发控制台，进入对应环境的云存储，检查是否还有残留文件。
6. 如果云存储还有文件，并且确认都是上线前测试资源，在控制台执行全量删除。
7. 回到数据库确认 `wdd-config` 仍存在，其他业务集合为空。
8. 删除或下线临时云函数 `wdd-temp-clear-all`。

## 为什么不能只靠云函数全量清空云存储

云函数里的 `wx-server-sdk` 目前支持按 `fileID` 删除文件，但没有提供“列出整个云存储目录/桶”的能力。没有文件清单，就无法在云函数里可靠删除所有文件。

所以这次上线前清理要分工：

- 云函数：清数据库，保留 `wdd-config` 和索引，删除数据库中引用过的云文件。
- 控制台：最终确认并清空云存储里已经失去数据库引用的残留文件。

## 使用后处理

这个云函数是临时运维工具。执行完成并确认无误后，建议从云环境中删除或下线 `wdd-temp-clear-all`，避免误调用。
