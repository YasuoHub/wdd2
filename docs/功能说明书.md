# 问当地小程序功能说明书

**更新日期**：2026-06-02  
**适用版本**：当前小程序开发版本  
**文档目标**：用产品语言说明”问当地”当前要交付给用户的功能范围和业务规则。

## 1. 产品定位

“问当地”解决的是用户无法亲临现场、但又需要快速获得某地实时信息的问题。例如：目的地是否下雨、前方路段是否拥堵、店铺是否营业、停车场是否还有空位、现场排队是否严重。

产品通过“地理位置匹配 + 悬赏激励 + 即时聊天”的方式，让求助者发布问题，让当地或附近的帮助者接单回答。

当前版本采用“任务大厅主动接单”模式：求助发布后进入大厅，帮助者自行浏览并接单。系统暂不做自动派单、人工派单、强制指派或按顺序轮询派发。

## 2. 用户角色

| 角色 | 说明 | 主要能力 |
| --- | --- | --- |
| 求助者 | 有异地实时信息需求的人 | 发布求助、选择支付方式、聊天沟通、确认完成、评价、举报/申诉 |
| 帮助者 | 能提供当地实时信息的人 | 浏览任务、接单、聊天回答、获得收益、评价、举报/申诉 |
| 客服/管理员 | 处理争议和异常工单的人 | 查看举报/申诉工单、补充材料、裁定任务、处理资金分配 |
| 运营 | 查看运营数据和审批资金的人 | 查看运营分析看板、审批大额提现申请 |

同一个普通用户可以同时是求助者和帮助者。

## 3. 页面与导航

底部导航包含 4 个入口：

| 导航 | 页面 | 说明 |
| --- | --- | --- |
| 首页 | `pages/index/index` | 展示产品入口、快捷发布、附近任务等核心信息 |
| 大厅 | `pages/task-hall/task-hall` | 浏览可接任务，按距离、类型、排序筛选 |
| 消息 | `pages/messages/messages` | 查看聊天会话和系统通知 |
| 我的 | `pages/my/my` | 查看个人资料、我的求助、我的任务、钱包、记录、设置 |

非底部导航页面包括发布求助、任务详情、聊天、钱包、提现、评价、举报、申诉、客服工单、公开资料、运营分析、资金审批等。

## 4. 用户与资料功能

### 4.1 微信登录

用户通过微信授权登录。系统根据微信 `openid` 识别用户身份，首次登录创建用户档案，并赠送新用户积分。

### 4.2 个人资料

用户可维护头像、昵称等基础信息。帮助者可配置帮助者资料，包括常去地点、帮助意愿和可帮助类型，用于提高任务匹配质量。

### 4.3 信用与评价

用户拥有信用分、平均评分、评价数量等信誉信息。任务完成后双方可以互评，评价会影响用户公开资料的可信度。

## 5. 求助发布功能

### 5.1 求助类型

当前支持的核心求助场景包括：

1. 实时天气
2. 道路拥堵
3. 店铺营业
4. 停车空位
5. 排队情况
6. 其他本地实时信息

### 5.2 发布内容

发布求助时，用户需要填写：

| 内容 | 说明 |
| --- | --- |
| 求助类型 | 选择最接近的问题类型 |
| 求助地点 | 可使用地图选点或手动补充地点名称 |
| 问题描述 | 描述想知道什么，建议具体、可回答 |
| 悬赏金额 | 设定任务赏金，范围 ¥1 ~ ¥500 |
| 支付方式 | 可选择微信支付或余额支付 |
| 有效期 | 支持 30 分钟、1 小时、2 小时、4 小时、12 小时、24 小时 |
| 图片 | 可上传现场、店铺、路线等辅助图片 |

### 5.3 支付方式

发布求助支持两种支付方式，用户可在发布页切换选择：

| 支付方式 | 说明 | 适用场景 |
| --- | --- | --- |
| 微信支付 | 调起微信收银台完成支付，金额直接从微信扣款 | 钱包余额不足时 |
| 余额支付 | 使用钱包可用余额支付，金额从余额中冻结 | 钱包有足够余额时 |

#### 余额支付机制

余额支付采用”先冻结、完成后再实际扣款”的机制：

1. **发布时**：从钱包可用余额中冻结悬赏金额（可用余额 = 总余额 - 已冻结金额）。余额数值不变，冻结金额增加。收支明细显示”余额支付冻结”。
2. **任务完成时**：解冻并实际扣款。余额减少，冻结金额减少。收支明细显示”任务支出”。
3. **任务取消/超时时**：解冻退回。冻结金额减少，余额数值不变（支付时未扣）。收支明细显示”余额退回”。

### 5.4 支付与发布

发布流程采用”先创建支付订单，再确认支付并创建任务”的方式：

- **微信支付**：拉起微信收银台 → 支付成功 → 任务进入 `pending` 待匹配状态
- **余额支付**：一步完成冻结余额 + 创建订单 + 创建任务 → 任务进入 `pending` 待匹配状态

支付成功后，任务出现在任务大厅。

## 6. 任务大厅与接单

### 6.1 任务浏览

帮助者可在任务大厅浏览待匹配任务，任务信息包含类型、地点、距离、悬赏、发布时间、有效期和描述摘要。

### 6.2 筛选与排序

当前支持按距离、类型、时间、悬赏等维度进行筛选或排序。距离计算依赖用户当前位置。

### 6.3 接单规则

帮助者点击接单后，系统会检查任务是否仍为待匹配状态。接单成功后：

1. 任务状态从 `pending` 变为 `ongoing`。
2. 创建接单记录。
3. 求助者和帮助者可进入聊天。
4. 系统向相关用户发送通知。

当前版本不包含派单功能。所谓“匹配”主要体现为大厅列表的距离、类型、排序和筛选，不代表系统会把任务自动分配给某个帮助者。

## 7. 聊天与消息

### 7.1 聊天范围

聊天围绕具体任务展开，只有任务相关的求助者和帮助者可以进入对应会话。

### 7.2 消息类型

当前支持文字和图片消息。消息会记录发送者、接收者、任务、内容、已读状态和发送时间。

### 7.3 实时提醒

小程序启动后会监听未读消息。收到新消息时，消息 Tab 会显示角标。若实时监听失败，聊天页可通过接口刷新消息列表。

### 7.4 暂不包含的消息管理能力

消息中心当前只承担会话和通知入口。消息置顶、左滑删除、会话归档等管理能力暂不进入发布前范围，后续等真实消息量上来后再评估。

## 8. 任务状态与结算

### 8.1 任务状态

| 状态 | 中文名 | 说明 |
| --- | --- | --- |
| `pending` | 待匹配 | 已发布，等待帮助者接单 |
| `ongoing` | 进行中 | 已有人接单，双方沟通中 |
| `completed` | 已完成 | 任务正常完成或被裁定完成 |
| `cancelled` | 已取消 | 任务取消、超时取消或被裁定取消 |
| `breaking` | 审核中 | 出现举报/申诉，等待客服裁定 |

### 8.2 正常完成

任务完成后，系统按平台配置计算服务费和帮助者收入，将帮助者收入记入钱包余额，并生成余额记录。

### 8.3 取消与超时

待匹配任务在有效期内无人接单或由发布者取消时，可进入取消流程。已支付订单需要配合退款流程处理。

进行中的任务超过规则时间后，可由系统自动完成或提醒用户处理，具体执行由自动任务云函数负责。

## 9. 钱包、余额与提现

### 9.1 钱包概览

钱包页展示用户的资金状况：

| 信息 | 说明 |
| --- | --- |
| 可用余额 | 可自由支配的金额（总余额 - 冻结金额） |
| 冻结金额 | 正在流程中的金额（提现申请冻结、余额支付冻结） |
| 累计收入 | 历史完成任务获得的总收入 |
| 累计提现 | 历史成功提现的总金额 |

### 9.2 冻结余额机制

冻结余额用于保障交易安全，防止金额被重复使用。以下场景会产生冻结：

| 场景 | 冻结时机 | 解冻时机 |
| --- | --- | --- |
| 余额支付发布任务 | 发布时冻结悬赏金额 | 任务完成时扣款 / 任务取消时退回 |
| 提现申请（大额） | 提交审批时冻结提现金额 | 审批驳回/过期时解冻 / 提现成功时扣减 |

冻结金额不可用于提现或新的余额支付。可用余额 = 余额 - 冻结金额。

### 9.3 收支明细

余额变动会写入 `wdd-balance-records`，在钱包页"收支明细"区域展示。记录类型包括：

| 类型 | 说明 | 方向 |
| --- | --- | --- |
| `task_income` | 任务收入（帮助者完成任务所得） | 收入（绿色） |
| `task_pay` | 任务支出（求助者余额支付扣款） | 支出（红色） |
| `refund` | 退款（任务取消/超时退回） | 收入（绿色） |
| `withdraw` | 提现（成功转出到微信） | 支出（红色） |
| `withdraw_fee` | 提现手续费 | 支出（橙色） |
| `freeze` | 资金冻结（余额支付/提现申请） | 标记（橙色） |
| `unfreeze` | 冻结解除（驳回/过期退回） | 收入（绿色） |
| `arbitration_refund` | 客服裁定退款 | 收入（绿色） |

> **注意**：提现失败不会在收支明细中产生任何记录（提现记录仅在转账成功后才写入）。

### 9.4 提现规则

用户可从钱包发起提现。系统根据提现金额与阈值的比较，自动选择提现路径：

| 条件 | 路径 | 说明 |
| --- | --- | --- |
| 金额 ≤ 提现阈值 | **即时提现** | 直接调用微信商家转账，立即到账 |
| 金额 > 提现阈值 | **审批提现** | 先提交申请等待管理员审批，审批通过后再手动发起转账 |

提现阈值由平台运营配置（`withdraw_approval_threshold`），可根据运营需要动态调整。

#### 单日提现限制

为了保护资金安全和满足合规要求，系统对提现设有每日限制：

| 限制项 | 默认值 | 配置字段 | 说明 |
| --- | --- | --- | --- |
| 单日次数上限 | 3 次 | `withdraw_daily_times` | 每日最多可发起的提现次数（含即时提现和审批提现） |
| 单日金额上限 | ¥5,000 | `withdraw_daily_limit` | 每日累计提现金额上限 |

> **注意**：单日次数和金额限制在即时提现和审批申请两个环节均会校验。当日已提现次数/金额包含所有状态为"处理中"、"已完成"、"打款中（重试）"的提现记录。

### 9.5 提现审批流程

大额提现（金额超过阈值）需要经过完整的审批流程：

```
用户提交申请 → 余额冻结 → 等待审批
                            ├── 审批通过 → 用户手动点击提现 → 微信转账 → 成功（余额扣除 + 收支记录）
                            │                                      └── 失败 → 余额恢复，可重试
                            ├── 审批驳回 → 余额解冻 → 收支显示"冻结解除"
                            └── 3天过期 → 余额解冻 → 收支显示"冻结解除"
```

**关键规则**：
- 审批通过后 3 天内有效，过期自动解冻
- 审批通过后由用户**手动**发起真实提现，不是自动到账
- 提现失败（如微信侧参数错误、余额不足等）不会改变申请状态，用户可再次点击提现重试
- 提现失败**不在收支明细中留痕**，失败信息写入 `wdd-withdraws` 审计记录
- 即时提现（小额）不经过审批流程，直接调起微信转账

### 9.6 钱包页面展示

除余额信息外，钱包页还展示：

- **提现申请列表**：展示用户的提现申请记录，含状态标签（等待审批 / 已通过 / 已驳回 / 已过期）、操作按钮（审批通过后显示"提现"按钮）
- **收支明细列表**：按时间倒序展示所有余额变动记录，每条显示图标、类型、金额（收入/支出颜色区分）、余额快照、相对时间

## 10. 积分、签到与邀请

积分用于平台内权益体系，不等同现金，不可提现。当前积分来源包括新用户赠送、签到奖励、邀请奖励等。积分记录写入 `wdd-point-records`，签到记录写入 `wdd-sign-in-records`。

## 11. 举报、申诉与客服工单

### 11.1 举报

当用户认为对方存在虚假信息、失联、诱导私下交易、辱骂骚扰、恶意拖延等行为时，可以发起举报。

举报创建后：

1. 写入 `wdd-reports`。
2. 创建客服工单 `wdd-tickets`。
3. 任务进入 `breaking` 审核中。
4. 用户可撤销举报或补充材料。

### 11.2 申诉

当用户对任务处理、金额结算、举报结果或对方行为有异议时，可以发起申诉。

申诉创建后：

1. 写入 `wdd-appeals`。
2. 创建客服工单 `wdd-tickets`。
3. 任务进入 `breaking` 审核中。
4. 用户可撤销申诉或补充材料。

### 11.3 客服裁定

客服可对工单进行裁定，结果包括：

| 裁定 | 任务结果 | 资金结果（微信支付） | 资金结果（余额支付） |
| --- | --- | --- | --- |
| 取消任务 | `cancelled` | 悬赏退回到求助者钱包余额 | 冻结金额解冻退回 |
| 完成任务 | `completed` | 悬赏结算给帮助者 | 解冻并实际扣款，结算给帮助者 |
| 部分完成 | `completed` | 按比例分配，退款部分退回钱包 | 已消费部分扣款，剩余解冻 |

## 12. 运营分析

### 12.1 功能概述

运营分析页面（`pages/ops-analytics/ops-analytics`）为运营人员提供关键业务数据的可视化看板，帮助监控平台健康度和增长趋势。页面入口位于"我的"页面，仅对运营角色可见。

### 12.2 分析指标

运营分析看板包含以下分析维度：

| 模块 | 说明 | 图表类型 |
| --- | --- | --- |
| 核心指标概览 | 用户总数、任务总数、完成率、GMV 等 | 数字卡片 |
| 用户增长趋势 | 每日新增用户数变化 | 折线图 |
| 任务发布趋势 | 每日发布任务数变化 | 折线图 |
| 任务类型分布 | 各求助类型的占比 | 饼图 |
| 任务转化漏斗 | 发布 → 匹配 → 完成 各阶段转化率 | 漏斗图 |
| 收入趋势 | 平台每日收入变化 | 柱状图 |

### 12.3 数据来源

运营分析数据由 `wdd-ops-analytics` 云函数提供，基于数据库集合进行聚合计算。

## 13. 资金审批

### 13.1 功能概述

资金审批页面（`pages/fund-approval/fund-approval`）为管理员提供提现申请的审批入口。管理员可在此页面查看所有待审批的提现申请，并进行通过或驳回操作。

### 13.2 审批操作

| 操作 | 说明 | 后续 |
| --- | --- | --- |
| 通过 | 批准该笔提现申请 | 用户端显示"已通过"，可在 3 天内手动点击提现 |
| 驳回 | 拒绝该笔提现申请 | 填写驳回理由，冻结金额退回用户可用余额 |

### 13.3 审批记录

审批操作会记录处理人、处理时间和处理结果。用户可在钱包页的提现申请列表中查看审批状态。

## 14. 自动任务与定时器

### 14.1 定时任务概述

`wdd-auto-cancel` 云函数作为统一的定时触发器入口，定期执行以下维护任务：

| 任务 | 说明 | 执行频率 |
| --- | --- | --- |
| 超时任务取消 | 扫描超时的待匹配任务，自动取消并退款 | 定时触发 |
| 进行中任务自动完成 | 超过规则时限的进行中任务自动完成 | 定时触发 |
| 提现重试 | 对 `transfer_pending` 状态的提现记录发起重试 | 定时触发 |
| 提现状态查询 | 查询 `processing` 状态超时的提现单在微信侧的状态 | 定时触发 |
| 审批申请过期 | 扫描已过期未提现的审批申请，自动解冻 | 定时触发 |

### 14.2 提现失败处理

提现失败分为两类处理：

| 失败类型 | 处理方式 | 用户感知 |
| --- | --- | --- |
| 明确业务错误（参数错误、余额不足等） | 立即回滚余额，`wdd-withdraws` 标记为 `rejected` | 提现申请不变，可重试；收支明细无痕 |
| 网络/系统错误 | 标记为 `transfer_pending`，排队等待定时器重试 | 无感知，后台自动重试 |
| 重试达上限 | 标记为 `transfer_failed`，需人工介入 | 不影响申请状态 |

## 15. 当前边界与后续增强

当前版本（2026-06-02）已完成以下核心能力：

1. ✅ 完整发布→匹配→接单→聊天→结算交易闭环
2. ✅ 微信支付 + 余额支付双支付模式
3. ✅ 冻结余额机制（支付冻结 + 提现冻结）
4. ✅ 大额提现审批流程
5. ✅ 客服工单争议处理（举报、申诉、裁定）
6. ✅ 运营分析看板
7. ✅ 收支明细完整展示
8. ✅ 定时器自动维护（超时取消、自动完成、提现重试、审批过期）

发布前必须确认的产品边界：

1. 帮助者通过任务大厅主动接单，当前没有自动派单功能。
2. 消息置顶、消息删除、会话归档暂不做，不作为发布前阻塞项。
3. 前端不得在接口失败时展示测试任务、测试用户或模拟业务数据；失败时应展示空状态、错误状态或重试入口。
4. 发布、支付、接单、聊天、完成、退款、提现、举报、申诉、客服裁定必须用真实数据完成回归测试。
5. 积分仅作为平台权益体系入口，不参与任务支付、结算、退款或提现。

后续计划增强：

1. 更精确的地理围栏和帮助者真实性验证。
2. 管理后台可视化，减少直接操作云函数和数据库。
3. 更完整的内容安全、图片审核和风控策略。
4. 任务推荐算法和帮助者等级体系。
5. 积分权益兑换功能。
6. 消息置顶、消息删除、会话归档等消息管理功能。
7. 派单或推荐接单机制，但必须在主动接单模式验证后再设计。
