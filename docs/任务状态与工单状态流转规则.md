# 众包小程序 最终版：任务状态 + 举报申诉工单 全流程设计
## 一、基础枚举（最终定稿）
### 1. 任务状态（5个）
```js
const STATUS_MAP = {
  // 仅任务被举报/申诉 → 平台介入处理时，才会进入【审核中】状态
  breaking: { text: '审核中', class: 'breaking', icon: '⏳' },
  pending: { text: '待匹配', class: 'pending', icon: '⏳' },  // 任务发布后默认状态
  ongoing: { text: '进行中', class: 'ongoing', icon: '🤝' },
  completed: { text: '已完成', class: 'completed', icon: '✅' },
  cancelled: { text: '已取消', class: 'cancelled', icon: '❌' }
};
```
### 2. 工单状态（3个）
- `pending`：未处理（客服待裁定）
- `resolved`：已处理（客服已裁定）
- `closed`：已关闭（用户撤销举报/申诉，工单自动关闭）

### 3. 举报/申诉类型
1. 发布者举报接单者
2. 接单者举报发布者
3. 双方对举报处理结果提起申诉

---

## 二、任务主状态流转（100%贴合你的规则）
### 核心流转主线（无举报场景）
**发布任务 → 待匹配(pending) → 进行中(ongoing) → 已完成(completed)**
### 核心分支（有举报/申诉场景）
进行中/已完成 **发起举报/申诉** → **审核中(breaking)** → 客服裁定 → 已完成/已取消
### 取消分支
待匹配 **发单人手动取消** → 已取消(cancelled)

---

## 三、每个状态 进入/流出时机（精准定义）
### 1. 待匹配 pending（任务初始状态）
- **进入时机**：发单人**成功发布任务**，直接进入待匹配
- **流出时机**：
  ① 用户接单 → 流转 **进行中(ongoing)**
  ② 发单人手动取消（唯一可取消权限） → 流转 **已取消(cancelled)**
- **规则**：不可举报、不可申诉

### 2. 进行中 ongoing（核心执行状态）
- **进入时机**：待匹配状态有人接单
- **流出时机**：
  ① 超时72小时 → 系统自动流转 **已完成(completed)**
  ② 主动确认完成 → 流转 **已完成(completed)**
  ③ **发起举报/申诉** → 流转 **审核中(breaking)**（平台介入）
- **超时规则**：超时24小时 → 仅通知发布者，状态**保持进行中**

### 3. 已完成 completed（终态之一）
- **进入时机**：
  ① 进行中72小时自动超时
  ② 双方主动确认完成
  ③ 审核中(breaking) 客服裁定完成/部分完成
- **流出时机**：**发起举报/申诉** → 流转 **审核中(breaking)**
- **规则**：允许发起举报/申诉

### 4. 审核中 breaking（平台介入临时状态）
- **【唯一进入时机】**：任务处于 **进行中/已完成**，用户发起**举报/申诉**，立即进入该状态
- **【唯一流出时机】**：客服完成工单裁定，强制流出
  ① 裁定取消任务 → 流转 **已取消(cancelled)**
  ② 裁定完成/部分完成 → 流转 **已完成(completed)**
- **规则**：双方均不可操作任务，等待客服裁定

### 5. 已取消 cancelled（最终终态）
- **进入时机**：
  ① 待匹配发单人手动取消
  ② 审核中(breaking) 客服裁定取消任务
- **流出时机**：无，永久终态（例外：客服裁定取消 `arbitration_cancelled` 时允许申诉）
- **规则**：禁止举报、常规操作；仅 `arbitration_cancelled` 裁定取消时允许申诉

## 四、角色操作权限表（一目了然）
| 任务状态 | 发单人可操作 | 接单者可操作 | 能否发起举报/申诉 |
|----------|--------------|--------------|------------------|
| 待匹配 | 取消任务 | 接单 | 否 |
| 进行中 | 举报接单者 | 举报发布者、执行任务 | 是 |
| 已完成 | 举报接单者、申诉 | 举报发布者、申诉 | 是 |
| 审核中 | 等待客服裁定 | 等待客服裁定 | 否 |
| 已取消 | 无 | 无 | 否 |

## 五、举报/申诉工单 状态流转 + 与任务联动
### 1. 工单独立流转（极简2步）
**用户发起工单** → 工单初始 `pending(未处理)` → 客服裁定 → 工单变为 `resolved(已处理)`
### 2. 工单与任务状态 强绑定时机
1. **发起举报/申诉瞬间**
   工单创建（pending） + 任务状态 **强制变为 breaking(审核中)**
2. **客服裁定瞬间**
   工单状态变为 resolved（已处理） + 任务状态 **从breaking流出到完成/取消**

### 3. 客服3种裁定结果（任务+资金同步生效）
| 裁定结果 | 任务状态变更 | 资金规则 |
|----------|--------------|----------|
| 取消任务 | breaking → cancelled | 奖励全额退回发单人 |
| 完成任务 | breaking → completed | 奖励全额发放接单者 |
| 部分完成 | breaking → completed | 奖励按比例分配双方 |

## 六、核心关系总结（开发必看）
1. **审核中(breaking) 专属用途**：**仅举报/申诉触发**，是平台处理工单的临时状态，和任务发布无关
2. **任务状态控制工单**：只有进行中、已完成可发起工单；审核中、待匹配、已取消禁止发起
3. **工单裁定控制任务**：审核中的任务，**只有客服裁定**能改变最终状态
4. **超时规则独立**：进行中72小时自动完成，不受工单影响；若已触发举报进入breaking，以客服裁定为准
5. **取消权限唯一**：只有待匹配状态，发单人可手动取消任务

---

### 总结
1. **核心提示**：`breaking(审核中)` 是**举报专属状态**，任务发布后直接是`pending(待匹配)`
2. **闭环逻辑**：举报触发审核中 → 客服裁定结案 → 任务落定完成/取消
3. **全规则落地**：超时通知、取消权限、工单状态、裁定资金
